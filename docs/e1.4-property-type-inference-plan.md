## Open Questions / Risks

1. **Scope of JSX traversal** — scanning the entire component tree for `<input>` elements would require a full recursive AST visitor. Scope is limited to: top-level statements → function declaration bodies → return statements → JSX tree depth. This covers the common Next.js single-component pattern; deeply nested JSX in child components is deferred.
2. **JSX hint → event association** — multiple form submit handlers may exist in one file. JSX input hints are merged into ALL `FormSubmit` events in the file (conservative over-attribution). Accurate per-handler association is deferred to a future improvement.
3. **FormalParameter.type_annotation** — in OXC 0.134, the TS type annotation for a function parameter is at `FormalParameter.type_annotation`, NOT on `FormalParameter.pattern`. Code must access `param.type_annotation`, not `param.pattern.type_annotation` (which doesn't exist in this version).
4. **BindingPattern enum** — `BindingPattern` is a direct enum (not a struct). Pattern match: `BindingPattern::BindingIdentifier(id)`. Destructured params (`ObjectPattern`, `ArrayPattern`) are skipped.
5. **FunctionBody.statements** — OXC 0.134 uses `FunctionBody.statements` (not `.body`). Code must use `.statements`.
6. **ArrowFunctionExpression.body** — for expression-body arrows (`x => expr`), `expression: bool` is `true` and the body's `statements` are empty. The plan handles block-body arrows only; expression-body arrows are skipped for JSX traversal.
7. **`type_from_name` conservatism** — only returns a type when highly confident (boolean/number patterns). `None` is correct when uncertain; callers may display "unknown" type. Does NOT attempt to infer "string" for general text fields — TS type annotations are the source of truth for typed params.
8. **`enrich_hints` idempotency** — safe to call multiple times. Existing `type_hint` values are preserved; PII flags can only go from `false` → `true`.

#### Public API

```rust
//! Property type inference and PII detection (E1.4).

use crate::{adapter::PropertyHint, namer::split_identifier};

// ─── PII detection ────────────────────────────────────────────────────────────

/// Tokens that, when found in a split-identifier property name, indicate PII.
const PII_TOKENS: &[&str] = &[
    "email", "mail", "phone", "mobile", "tel",
    "address", "addr", "street", "city", "zip", "postal", "postcode", "state", "country",
    "name", "firstname", "lastname", "surname", "givenname", "familyname",
    "dob", "birthdate", "birthday", "birth",
    "ssn", "social", "national", "passport", "license",
    "credit", "card", "cvv", "cvc",
    "token", "secret", "password", "passwd", "pin",
    "ip",
];

/// Returns `true` when `name` contains a PII token after splitting on camelCase/snake_case/kebab.
pub fn is_pii_property(name: &str) -> bool {
    split_identifier(name).iter().any(|t| PII_TOKENS.contains(&t.as_str()))
}
```

**Important:** `name` is not compared directly against `PII_TOKENS` — it's split into tokens first.
- `"email"` → tokens `["email"]` → true
- `"userEmail"` → tokens `["user", "email"]` → true ("email" in PII_TOKENS)
- `"emailAddress"` → tokens `["email", "address"]` → true
- `"method"` → tokens `["method"]` → false
- `"endpoint"` → false

#### `pub fn type_from_name(name: &str) -> Option<&'static str>`

Heuristic JS/TS type inference from property name tokens.

```rust
pub fn type_from_name(name: &str) -> Option<&'static str> {
    let tokens = split_identifier(name);
    let first = tokens.first().map(String::as_str).unwrap_or("");
    let last  = tokens.last().map(String::as_str).unwrap_or("");

    // Boolean patterns: first token is a boolean-prefix word
    if matches!(first, "is" | "has" | "can" | "should" | "was" | "did" | "will") {
        return Some("boolean");
    }
    // Boolean patterns: last token is a boolean-suffix word
    if matches!(last, "enabled" | "disabled" | "active" | "checked" | "visible" | "hidden" | "flag") {
        return Some("boolean");
    }

    // Number patterns: last token is an unambiguously numeric concept
    if matches!(
        last,
        "count" | "total" | "amount" | "price" | "quantity" | "age"
        | "size" | "limit" | "offset" | "page" | "index" | "rank"
        | "score" | "weight" | "height" | "width" | "length" | "duration"
    ) {
        return Some("number");
    }
    // Timestamp names are numbers (Unix timestamp convention)
    if matches!(name, "timestamp" | "created_at" | "updated_at" | "deleted_at") {
        return Some("number");
    }

    None
}
```

Returns `None` when no confident pattern matches (conservative). Callers treat `None` as "type unknown".

#### `pub fn enrich_hints(hints: Vec<PropertyHint>) -> Vec<PropertyHint>`

```rust
pub fn enrich_hints(mut hints: Vec<PropertyHint>) -> Vec<PropertyHint> {
    for hint in &mut hints {
        if hint.type_hint.is_none() {
            if let Some(t) = type_from_name(&hint.name) {
                hint.type_hint = Some(t.to_owned());
            }
        }
        if !hint.pii_hint {
            hint.pii_hint = is_pii_property(&hint.name);
        }
    }
    hints
}
```

Invariants:
- Existing `type_hint` values are never overwritten.
- `pii_hint` can only change `false` → `true`.
- Idempotent: calling twice yields the same result.

#### `pub(crate) fn hints_from_params<'a>(params: &FormalParameters<'a>) -> Vec<PropertyHint>`

Extract property hints from OXC function parameters.

```rust
pub(crate) fn hints_from_params<'a>(
    params: &oxc_ast::ast::FormalParameters<'a>,
) -> Vec<PropertyHint> {
    use oxc_ast::ast::{BindingPattern, TSType};

    params.items.iter().filter_map(|p| {
        // Only handle simple identifier params; skip destructured params.
        let BindingPattern::BindingIdentifier(id) = &p.pattern else { return None; };
        let name = id.name.to_string();

        // Extract TS primitive type from type annotation when present.
        let type_hint = p.type_annotation.as_ref().and_then(|ta| {
            match &ta.type_annotation {
                TSType::TSStringKeyword(_)  => Some("string".to_owned()),
                TSType::TSNumberKeyword(_)  => Some("number".to_owned()),
                TSType::TSBooleanKeyword(_) => Some("boolean".to_owned()),
                _ => None,
            }
        });

        Some(PropertyHint { name, type_hint, pii_hint: false })
    }).collect()
}
```

Notes:
- `p.type_annotation` is on `FormalParameter`, NOT on `BindingPattern` (OXC 0.134 specific).
- Destructured params (`ObjectPattern`, `ArrayPattern`) return `None` → skipped.
- Non-primitive TS types (interfaces, generics, `any`) yield `type_hint = None`.
- `pii_hint` is `false` here; callers apply `enrich_hints` afterwards.

#### `pub(crate) fn hints_from_jsx_inputs<'a>(prog: &Program<'a>) -> Vec<PropertyHint>`

Scan JSX in the program for `<input>`, `<select>`, `<textarea>` elements and extract their `name` attribute.

Walk strategy:
1. For each top-level statement:
   - `FunctionDeclaration` → walk its body statements
   - `ReturnStatement` → walk the return expression
   - `ExportDefaultDeclaration(FunctionDeclaration | Expression)` → walk body/expression
   - `ExportNamedDeclaration(FunctionDeclaration | VariableDeclaration)` → walk body/init
   - `VariableDeclaration` → walk each declarator's init
   - `ExpressionStatement` → walk the expression
2. For each expression:
   - `JSXElement` → recurse into `collect_from_jsx`
   - `JSXFragment` → recurse into children
   - `ParenthesizedExpression` → recurse into inner expression
   - `ArrowFunctionExpression` (block body only) → walk body statements
3. For each JSX element:
   - If tag is `"input"` / `"select"` / `"textarea"` AND has `name` attribute as string literal → add hint
   - Recurse into `children: Vec<JSXChild>`: visit `Element` and `Fragment` variants

Deduplicate by name (first occurrence wins).

```
fn collect_from_stmt(stmt: &Statement<'_>, hints: &mut Vec<PropertyHint>) { ... }
fn collect_from_expr(expr: &Expression<'_>, hints: &mut Vec<PropertyHint>) { ... }
fn collect_from_jsx(elem: &JSXElement<'_>, hints: &mut Vec<PropertyHint>) { ... }
fn collect_from_jsx_child(child: &JSXChild<'_>, hints: &mut Vec<PropertyHint>) { ... }
fn extract_name_attr(elem: &JSXOpeningElement<'_>) -> Option<String> { ... }
```

`hints_from_jsx_inputs` result has `type_hint = None`, `pii_hint = false`; callers apply `enrich_hints`.

## Review Summary

- **Iterations run:** 1
- **Gate status:** All files ≥ 96%. Overall 97.3% ≥ 96%.
- **Key design decision:** `hints_from_params` is `pub(crate)` because its signature contains `oxc_ast::ast::FormalParameters` — a type from an internal dependency. External callers don't have access to OXC AST nodes, so public export would be unusable and confusing.
- **Key design decision:** `enrich_hints` is applied once at the end of `analyze()` rather than inline at each site. This ensures consistent enrichment regardless of which detection path produced the properties.
- **Key design decision:** JSX hint → event association is "all form events in the file" rather than per-component. This is over-attributive but safe; false positives are clearly labelled as `Proposed` and humans can remove them.
- **Backward compat:** No existing types change; `enrich_hints` can only add data (PII flags, missing types). All 248 existing tests must continue to pass.
