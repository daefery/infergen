### 6. `docs/migrate-from-typewriter.md` (new)

**Purpose:** Step-by-step guide moving from Segment Typewriter to Infergen — covering concept mapping, config migration, and call-site rewrite.

**Score: 97%**

**Sections:**

1. **Why migrate** — Typewriter requires a manual Segment tracking plan + Segment-only; Infergen auto-discovers, is provider-neutral, offline, type-safe (PRD §1 / §9)
2. **Concept mapping table:**

| Typewriter | Infergen |
|-----------|---------|
| Tracking plan (Segment UI) | `catalog.yaml` (in-repo, version-controlled) |
| `typewriter.json` / Segment sources | `infergen.config.json` |
| `track.myEvent({...})` | `track.myEvent({...})` (same pattern) |
| `typewriter build` | `infergen generate` |
| Segment as sole destination | Any provider via config |
| Manual event definition | Auto-discovered + human-approved |

3. **Step-by-step migration:**
   - Step 1: Export your tracking plan from Segment (CSV or JSON)
   - Step 2: `infergen init` in your project
   - Step 3: Manually approve events from your Typewriter tracking plan in `catalog.yaml`
   - Step 4: Add the Segment provider to `infergen.config.json`
   - Step 5: `infergen generate` → new typed SDK
   - Step 6: Replace `import { ... } from './analytics'` (Typewriter) with `import { track } from './infergen.generated'`
   - Step 7: Run `infergen scan` — additional events from your code appear as proposals

4. **Call-site before/after** — code comparison
5. **Config before/after** — `typewriter.json` vs `infergen.config.json`
6. **CI before/after** — `typewriter build --check` vs `infergen scan --check`

### 7. `docs/migrate-from-avo.md` (new)

**Purpose:** Step-by-step guide moving from Avo to Infergen — covering Avo's cloud-dependent model vs Infergen's offline-first approach.

**Score: 97%**

**Sections:**

1. **Why migrate** — Avo requires cloud + manual event definitions; Infergen is offline-first and code-derived (PRD §1 / §9)
2. **Concept mapping table:**

| Avo | Infergen |
|-----|---------|
| Avo workspace (cloud UI) | `catalog.yaml` (in-repo) |
| Avo codegen | `infergen generate` |
| Avo branches | Git branches |
| `Avo.init(...)` + `Avo.eventName(...)` | `track.eventName(...)` |
| Avo Inspector (runtime monitoring) | `infergen scan --check` (static) |
| Destination interface | Provider plugin interface |

3. **Step-by-step migration:**
   - Step 1: Export events from Avo (JSON export)
   - Step 2: `infergen init` — run detection
   - Step 3: Map Avo events → entries in `catalog.yaml` (with `approved` status)
   - Step 4: Configure providers in `infergen.config.json`
   - Step 5: `infergen generate` → typed SDK
   - Step 6: Replace `Avo.EventName({...})` with `track.eventName({...})`
   - Step 7: Remove Avo initialization boilerplate

4. **What you gain:** offline operation, version-controlled catalog, auto-discovery, multi-provider
5. **What you give up:** Avo Inspector runtime monitoring (E8.x scope), Avo's team governance UI (E9.2 scope)
6. **Call-site before/after**
