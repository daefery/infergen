# E8.2 — Test Suite & Fixtures

**Epic:** E8.2 — Test Suite & Fixtures  
**Milestone:** 8 — Hardening & Release  
**PRD refs:** §7 (NFR: Offline, Performance, Extensibility), §6.1–6.5  
**ROADMAP description:** "Golden-file tests per adapter, real-world fixture repos, codegen snapshot tests, runtime delivery tests."  
**Dependencies:** Cross-cutting; no hard blockers.  
**Status:** Planned → In Progress

---

## Context

Existing test suite (265 tests) uses:
- Inline source strings as fixtures (no on-disk fixture files)
- `assert!(output.contains(...))` for codegen (no full-output snapshots)
- Unit-level provider tests (no end-to-end pipeline delivery)

E8.2 adds:
1. **Real fixture files** on disk (`tests/fixtures/`) — realistic per-adapter source code and a rich catalog
2. **Golden-file adapter tests** (`tests/golden.rs`) — load fixtures, assert full structured event output
3. **Codegen snapshot tests** (`tests/codegen_snapshots.rs`) — capture complete generated code with `insta`
4. **End-to-end delivery tests** (`tests/delivery.rs`) — proposals → catalog → provider registry dispatch

---

## Build / Sequence Order

1. Fixture source files (`tests/fixtures/{framework}/…`)
2. Fixture catalog YAML (`tests/fixtures/catalog/e2e_catalog.yaml`)
3. Workspace `Cargo.toml` — add `insta` to workspace deps
4. `crates/infergen-core/Cargo.toml` — add `insta` to dev-deps
5. `crates/infergen-core/tests/golden.rs` — adapter golden tests
6. `crates/infergen-core/tests/codegen_snapshots.rs` — snapshot tests
7. `crates/infergen-core/tests/delivery.rs` — end-to-end delivery tests

interface Props {
  user: { id: string; email: string } | null;
}

export default function HomePage({ user }: Props) {
  const handleSubscribe = async (e: React.FormEvent) => {
    e.preventDefault();
    const target = e.target as typeof e.target & { email: { value: string } };

      <form onSubmit={handleSubscribe}>
        <input name="email" type="email" required />
        <button type="submit">Subscribe</button>
      </form>
    </main>
  );
}

export const getServerSideProps: GetServerSideProps = async (ctx) => {
  const session = await getSession(ctx);
  return { props: { user: session?.user ?? null } };
};

### 10. `crates/infergen-core/tests/golden.rs`

**Score: 96%**

#### Pattern

```rust
fn fixture(path: &str) -> String {
    std::fs::read_to_string(
        std::path::Path::new(env!("CARGO_MANIFEST_DIR"))
            .join("tests/fixtures")
            .join(path),
    )
    .unwrap_or_else(|_| panic!("fixture not found: {path}"))
}
```

This uses `env!("CARGO_MANIFEST_DIR")` (resolved at compile time to the crate root) — the standard Rust approach for fixture loading. Paths are relative to `crates/infergen-core/`.

### 12. `crates/infergen-core/tests/delivery.rs`

**Purpose:** End-to-end delivery tests: proposals → catalog → provider registry dispatch.  
**Score: 97%**

These tests verify the full pipeline from adapter output through catalog formation to provider delivery — testing that the system works as a whole, not just its parts in isolation.

#### Fixtures

```rust
fn mock_event(name: &str) -> TrackEvent { ... }  // same pattern as provider.rs

struct RecordingProvider { ... }  // copy from provider.rs (no shared fixture crate yet)
struct ArcPlugin(Arc<RecordingProvider>); // same wrapper
```

Note: copy the fixture structs from `provider.rs` rather than sharing — there is no `tests/common.rs` pattern in the existing suite, and adding one isn't in scope for E8.2. Cross-file fixture sharing is E8.5 scope.

#### Test cases

**`e2e_proposals_to_delivery`**

Full pipeline test:
1. Create 3 `ProposedEvent` objects (simulate adapter output)
2. Call `from_proposals()` → `Catalog`  
3. `approve_all_proposed()` → approve all
4. Extract event names from catalog
5. Build `TrackEvent` objects and dispatch through `ProviderRegistry`
6. Assert all 3 event names were received by `RecordingProvider`

```rust
let proposals = vec![
    ProposedEvent::new("page_viewed", EventKind::PageView, PathBuf::from("index.tsx"), 0.9),
    ProposedEvent::new("user_signed_in", EventKind::AuthEvent, PathBuf::from("auth.ts"), 0.85),
    ProposedEvent::new("checkout_completed", EventKind::ApiCall, PathBuf::from("checkout.ts"), 0.9),
];
let mut catalog = from_proposals(&proposals);
approve_all_proposed(&mut catalog);

let mut registry = ProviderRegistry::new();
let recorder = RecordingProvider::new("test");
registry.register(Box::new(ArcPlugin(Arc::clone(&recorder))));

for entry in catalog.events.iter().filter(|e| e.status == EventStatus::Approved) {
    registry.track(TrackEvent {
        name: entry.name.clone(),
        properties: HashMap::new(),
    }).unwrap();
}

let recorded = recorder.recorded();
assert_eq!(recorded.len(), 3);
assert!(recorded.contains(&"page_viewed".to_string()));
assert!(recorded.contains(&"user_signed_in".to_string()));
assert!(recorded.contains(&"checkout_completed".to_string()));
```

**`delivery_properties_passed_through`**

Verifies that properties from `TrackEvent` reach the provider unchanged:
- Create event with 3 properties (user_id string, price number as JSON, is_new_user bool as JSON)
- Dispatch through registry
- Provider checks `event.properties` map (extend `RecordingProvider` to store full events)

```rust
struct FullRecordingProvider { events: Mutex<Vec<TrackEvent>>, ... }
// Stores the full event, not just the name.
// Asserts: event.properties["user_id"] == "u123"
```

**`delivery_ordering_preserved`**

Dispatching N events in order → provider receives them in the same order:
- Send `["a", "b", "c", "d", "e"]` to registry
- Assert `recorder.recorded() == ["a", "b", "c", "d", "e"]`

**`delivery_multi_provider_each_receives_all`**

5 events, 3 providers:
- Each provider receives all 5 events
- Assertion: `p1.recorded().len() == 5 && p2.recorded().len() == 5 && p3.recorded().len() == 5`

**`delivery_partial_failure_isolation`**

3 providers: good1, failing, good2 — in that order:
- Dispatch 1 event
- Registry returns Err (from failing provider)
- `good1.recorded().len() == 1` ✓
- `good2.recorded().len() == 1` ✓ (continuing after failure)
- Error message identifies "failing" as the failing provider

**`delivery_flush_after_batch`**

After tracking 5 events, call `registry.flush()`:
- `recorder.flush_count() == 1`
- `recorder.recorded().len() == 5` (all events before flush)

**`delivery_shutdown_lifecycle`**

Full lifecycle:
1. Register provider
2. Track 3 events
3. Flush
4. Shutdown
- All 3 events received
- `flush_count == 1`
- `shutdown_count == 1`

## Open questions / risks

1. **`insta` snapshot acceptance workflow:** The initial run creates `.snap.new` files. Implementation must run `cargo insta accept` (or `INSTA_UPDATE=new cargo test` + `cargo insta review`) before committing. The `.snap` files must be committed to the repo.
2. **Fixture detection counts:** Adapter detection counts are stated as "≥ N" rather than exact because minor heuristic changes could alter counts. If a future adapter change legitimately reduces detection, the golden test needs updating — this is correct behavior for a golden test.
3. **Codegen APIs for Go/Python/Ruby:** `GoCodegenConfig` exists, but `CodegenConfig` is shared for Python. Ruby has `RubyCodegenConfig`. Need to verify exact API names in impl phase (the plan uses best-guess names; adjust if needed).
4. **`ProposedEvent::new` signature:** The delivery test uses a constructor — verify the actual signature in impl phase.
