**Epic:** E6.2

**PRD refs:** §6.1 (Discovery/Scanning), §8 (Architecture)

**Date:** 2026-06-09

---

## ⚠ Blocked-by Warning

**E6.1 (Local LLM Integration / Ollama) is `Planned` — not yet implemented.**

E6.2 lists E6.1 as a dependency because the flow detector can optionally invoke an LLM pass to refine groupings on ambiguous events. However, E6.1 is described as "optional path — never required" (ROADMAP) and "optional local-LLM pass" (PRD §6.1). The core flow-detection value — linking related events across files via route and name heuristics — does **not** require LLM.

**Resolution:** Implement E6.2 fully with heuristic-only detection. Add a `SemanticRefiner` trait as the LLM plug-in point so E6.1 can wire in without changing E6.2 code.

## Open Questions / Risks

- **Overlap between route and name groupers** — same flow detected by both; dedup strategy uses sorted event-name sets as keys.
- **E6.1 not available** — NoOpRefiner is the safe default; no correctness regression.
- **`CATALOG_SCHEMA_VERSION` bump** — adding `flows` field to Catalog YAML. Since the field is `#[serde(default, skip_serializing_if = "Vec::is_empty")]`, existing catalogs deserialize fine and the field is omitted when empty. No version bump required for backward compatibility.
- **Step ordering** — temporal suffix heuristic may mis-order unusual names. Flows with no ordering signal sort steps alphabetically; reviewers can reorder.

use crate::ProposedEvent;

/// A group of proposals sharing a common route prefix.
pub struct RouteGroup {

pub fn extract_route_segment(source_path: &Path) -> Option<String> { ... }

///

pub fn group_by_route_prefix(proposals: &[ProposedEvent]) -> Vec<RouteGroup> { ... }

Implementation of `extract_route_segment`:

### 4. `crates/infergen-core/src/flow/name_grouper.rs` (new)

**Purpose:** Group proposals by shared name prefix and infer step order from temporal suffix markers.

**Detailed spec:**

```rust
use crate::ProposedEvent;

/// Temporal ordering rank for name suffixes (lower = earlier in funnel).
/// Suffixes are matched as substrings of the final name segment.
const TEMPORAL_RANKS: &[(&str, u32)] = &[
    ("start",    0),
    ("initiat",  0),
    ("begin",    0),
    ("open",     1),
    ("view",     2),
    ("loaded",   2),
    ("enter",    3),
    ("submit",   4),
    ("complet",  5),
    ("confirm",  5),
    ("success",  6),
    ("done",     6),
    ("fail",     99),
    ("cancel",   99),
    ("abandon",  99),
    ("error",    99),
];

/// A group of proposals sharing a common name prefix.
pub struct NameGroup {
    /// The shared prefix, e.g. `"checkout"`.
    pub prefix: String,
    /// Proposal indices in temporal order. If no temporal marker detected,
    /// sorted lexicographically.
    pub ordered_indices: Vec<usize>,
    /// True when at least one proposal had a recognisable temporal suffix.
    pub has_order_signal: bool,
}

/// Extract the leading prefix from an event name by splitting on `_` and
/// stripping the last segment if it is a temporal marker.
///
/// `"checkout_submitted"` → `Some("checkout")`
/// `"checkout_step_submitted"` → `Some("checkout_step")`
/// `"page_viewed"` → `Some("page")` (single segment after strip)
/// `"clicked"` → `None` (no prefix remains)
pub fn name_prefix(name: &str) -> Option<String> { ... }

/// Assign a temporal rank to an event name (lower = earlier step).
pub fn temporal_rank(name: &str) -> u32 { ... }

/// Group proposals by common name prefix, ordered by temporal rank.
///
/// Only groups with ≥ 2 proposals are returned.
pub fn group_by_name_prefix(proposals: &[ProposedEvent]) -> Vec<NameGroup> { ... }
```

Implementation of `name_prefix`:
- Split name by `_`
- Check if the **last** segment matches any `TEMPORAL_RANKS` prefix
- If yes, drop the last segment and join the rest as prefix
- If prefix is empty (single segment) → return `None`
- If no temporal marker: check if name has ≥ 2 `_`-separated parts; take all-but-last as prefix

Implementation of `group_by_name_prefix`:
- Build `HashMap<String, Vec<usize>>` keyed by `name_prefix`
- For each group with ≥ 2 proposals:
  - Sort by `temporal_rank(name)` ascending
  - If all ranks are 0 (default), sort lexicographically
  - Build `NameGroup { prefix, ordered_indices, has_order_signal }`
- Return sorted by prefix

**Edge cases:**
- Names with no `_` separator → return `None` from `name_prefix` (excluded).
- All proposals have same rank → lexicographic fallback.
- Prefix collision between route group and name group → dedup in `FlowDetector`.

**Tests:**
- `checkout_steps_ordered` — `["checkout_started", "checkout_submitted", "checkout_completed"]` → prefix `"checkout"`, indices in start→submit→complete order.
- `single_name_no_group` — only one `checkout_*` event → no group.
- `names_without_temporal_sorted_alpha` — `["signup_email", "signup_phone"]` → group sorted alpha.
- `name_prefix_returns_none_for_single_segment` — `"clicked"` → `None`.
- `temporal_rank_fail_is_99` — `"checkout_failed"` → rank 99.

**Score: 97%**

### 6. `crates/infergen-core/src/flow/mod.rs` (new)

**Purpose:** `FlowDetector` orchestrates route grouper + name grouper + pattern matcher, deduplicates, assigns confidence, and calls the refiner.

**Detailed spec:**

```rust
use std::collections::HashMap;
use std::path::Path;

use infergen_types::FlowKind;

use crate::ProposedEvent;
use self::patterns::{KNOWN_PATTERNS, match_known_patterns};
use self::refiner::{NoOpRefiner, SemanticRefiner};
use self::route_grouper::group_by_route_prefix;
use self::name_grouper::group_by_name_prefix;

pub mod name_grouper;
pub mod patterns;
pub mod refiner;
pub mod route_grouper;

/// A step in a detected flow, referencing a proposal by index.
#[derive(Debug, Clone)]
pub struct DetectedStep {
    pub proposal_idx: usize,
    /// Zero-based step index within the flow.
    pub step_index: u32,
}

/// A funnel detected across proposals, before catalog assignment.
#[derive(Debug, Clone)]
pub struct DetectedFlow {
    /// Human-readable name derived from prefix or pattern.
    pub name: String,
    pub kind: FlowKind,
    /// Detection confidence: pattern match 0.85 · route 0.75 · name-only 0.60.
    pub confidence: f32,
    pub steps: Vec<DetectedStep>,
}

/// Detects multi-step funnels from a flat list of proposed events.
///
/// Algorithm:
/// 1. `route_grouper` — group proposals by common source-path prefix.
/// 2. `name_grouper` — group proposals by shared event-name prefix + temporal order.
/// 3. `patterns` — overlay known funnel templates (checkout, onboarding …).
/// 4. Merge overlapping groups: two groups covering the same proposal set are
///    unified into one flow (pattern name wins over generic prefix).
/// 5. Assign confidence: pattern-matched ≥ 0.85, route ≥ 0.75, name-only ≥ 0.60.
/// 6. Call `refiner.refine()` for optional LLM pass.
/// 7. Return flows with ≥ 2 steps and confidence ≥ 0.5.
pub struct FlowDetector {
    refiner: Box<dyn SemanticRefiner>,
}

impl Default for FlowDetector {
    fn default() -> Self {
        Self { refiner: Box::new(NoOpRefiner) }
    }
}

impl FlowDetector {
    pub fn new() -> Self { Self::default() }

    pub fn with_refiner(refiner: Box<dyn SemanticRefiner>) -> Self { Self { refiner } }

    /// Detect flows from `proposals`. Does not require `project_root` because
    /// source paths on `ProposedEvent` are already absolute; relative
    /// conversion is handled inside groupers.
    pub fn detect(&self, proposals: &[ProposedEvent]) -> Vec<DetectedFlow> { ... }
}
```

Implementation of `detect`:
1. Collect route groups: `group_by_route_prefix(proposals)` → candidate flows with confidence 0.75, kind `FlowKind::Custom`, name from prefix.
2. Collect name groups: `group_by_name_prefix(proposals)` → candidate flows with confidence 0.60, kind `FlowKind::Custom`, name from prefix.
3. Run `match_known_patterns(proposals)` → per-pattern `HashMap<pattern_idx, Vec<usize>>`; for patterns with ≥ min_matches hits, build a candidate flow with confidence 0.85 and the pattern's kind + name.
4. Merge candidates: key = sorted `Vec<usize>` of proposal indices. For each key, keep the highest-confidence candidate. If a pattern-match candidate overlaps ≥ 80% with a route/name candidate, the pattern wins.
5. Assign step indices from name grouper ordering where available; otherwise preserve route ordering; otherwise sort by source path.
6. Call `self.refiner.refine(&mut flows, proposals)`.
7. Filter: `confidence >= 0.5 && steps.len() >= 2`.
8. Sort by name for deterministic output.

**Dependencies:** all submodules above, `infergen-types`, `crate::ProposedEvent`.

**Tests:**
- `detect_checkout_flow_from_route` — proposals with `pages/checkout/*.tsx` → flow name `"checkout"`, kind `Checkout`.
- `detect_onboarding_flow_from_names` — proposals `["onboarding_started", "onboarding_profile_completed"]` → onboarding flow.
- `pattern_beats_generic_for_overlapping_group` — checkout pattern + route group overlap → single flow, kind `Checkout`, confidence ≥ 0.85.
- `single_event_per_prefix_no_flow` — only one file under `payment/` → no flow.
- `noop_refiner_does_not_alter_detected_flows` — refiner call produces same result.
- `detect_empty_returns_empty` — `[]` proposals → `[]` flows.
- `flows_sorted_by_name` — multiple flows → deterministic name order.

**Score: 97%**

**Purpose:** Add `assign_flows` that maps `DetectedFlow`s onto a `Catalog` and `rescan_merge` preserves existing flows.

Add after the existing `rescan_merge` function:

```rust
use crate::flow::{DetectedFlow, DetectedStep};
use infergen_types::{EventFlow, FlowKind, FlowStep};

/// Assign detected flows to a catalog.
///
/// For each `DetectedFlow`:
/// 1. Resolve each `DetectedStep.proposal_idx` → `ProposedEvent` → stable ID in `catalog`.
/// 2. Build `EventFlow { id, name, kind, steps, confidence }`.
/// 3. Set `flow_ids` back-reference on matching `CatalogEntry` rows.
/// 4. Replace `catalog.flows` with the newly assigned flows.
///
/// Existing manually-added flows (those with IDs not present in `detected`)
/// are preserved. Detected flows that match an existing flow ID are updated
/// in-place (steps may change; name/description edits are preserved).
pub fn assign_flows(
    catalog: &mut Catalog,
    detected: &[DetectedFlow],
    proposals: &[ProposedEvent],
    project_root: &Path,
) { ... }

/// Generate a stable flow ID from name and kind.
fn generate_flow_id(name: &str, kind: &str) -> String {
    // FNV-1a same scheme as event IDs, prefix "flow_"
    const FNV_OFFSET: u64 = 0xcbf2_9ce4_8422_2325;
    const FNV_PRIME:  u64 = 0x0000_0100_0000_01b3;
    let mut hash = FNV_OFFSET;
    for byte in format!("{name}:{kind}").bytes() {
        hash ^= u64::from(byte);
        hash = hash.wrapping_mul(FNV_PRIME);
    }
    format!("flow_{hash:016x}")
}
```

Implementation of `assign_flows`:
1. Build a lookup `HashMap<String, usize>` from (name + rel_path) → catalog index, using the same `generate_stable_id` logic.
2. For each `DetectedFlow`:
   - Generate `flow_id = generate_flow_id(name, kind_str)`.
   - For each `DetectedStep`, look up the proposal's stable event ID.
   - Build `FlowStep { event_id, step_index, optional: false }`.
   - Build `EventFlow { id: flow_id, name, kind, description: "".into(), steps, confidence: detected.confidence as f64 }`.
3. For `catalog.flows`:
   - Keep existing flows whose IDs are NOT in the detected set (manual flows).
   - For flows whose IDs ARE in detected, update steps but preserve `description` if non-empty.
   - Append new detected flows.
4. For each `CatalogEntry`, recompute `flow_ids` from the updated `catalog.flows`.

Update `rescan_merge` signature to accept optional flows:
- `rescan_merge` currently returns `Catalog`; the scan command calls `assign_flows` after.
- No signature change needed — `assign_flows` is called from `scan.rs` after `rescan_merge`.

**Dependencies:** `crate::flow::{DetectedFlow}`, `infergen_types::{EventFlow, FlowStep, FlowKind}`.

**Tests:**
- `assign_flows_populates_catalog_flows` — two-step checkout flow → catalog has 1 `EventFlow` with 2 `FlowStep`s.
- `assign_flows_sets_back_references` — matching `CatalogEntry.flow_ids` contains the flow ID.
