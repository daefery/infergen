## Open Questions / Risks

1. **Stable ID collision** — FNV-1a 64-bit over `{name}:{source_path}:{kind}` produces a 16-hex-digit ID. Collision probability for a 1,000-event catalog is negligible (~5×10⁻¹¹). If two distinct events have identical name+path+kind (impossible given ProposedEvent semantics), the second is deduplicated by `merge_proposals`.
2. **f32 YAML round-trip** — serde_yaml uses f64 internally. Deserializing back to `f32` may produce minor precision drift (e.g. `0.85` → `0.8500000238418579`). This is harmless for confidence scores but must be acknowledged. Mitigated: serialize confidence with `#[serde(serialize_with = "serialize_f32_2dp")]` to clamp to 2 decimal places.
3. **infergen-types gains serde dependency** — The crate doc says "dependency-free"; the `lib.rs` comment explicitly defers catalog types to E1.1. Adding `serde` breaks the zero-external-dep invariant. This is intentional and documented in E1.1. All downstream crates (`infergen-core`, `infergen-cli`) already depend on serde transitively — no new transitive cost.
4. **ProposedEvent has no adapter field** — E1.1 maps `ProposedEvent → CatalogEntry` without knowing which adapter produced it. `EventProvenance.adapter` will be empty string `""` until E1.2 adds adapter attribution. This is a known gap, not a bug.
5. **YAML key ordering** — `serde_yaml` (0.9.x) serializes struct fields in declaration order, which is deterministic. The plan spec declares fields in the order they should appear in the file. Events are sorted by `id` before save.

#### `generate_stable_id(name, source_path, kind) -> String`

```rust
fn generate_stable_id(name: &str, source_path: &str, kind: &str) -> String {
    // FNV-1a 64-bit, portable and deterministic across platforms/Rust versions.
    const FNV_OFFSET: u64 = 0xcbf2_9ce4_8422_2325;
    const FNV_PRIME: u64 = 0x0000_0100_0000_01b3;
    let mut hash = FNV_OFFSET;
    for byte in format!("{name}:{source_path}:{kind}").bytes() {
        hash ^= u64::from(byte);
        hash = hash.wrapping_mul(FNV_PRIME);
    }
    format!("evt_{hash:016x}")
}
```

Private helper. Produces `"evt_{016lowercase-hex}"` — 20-char string, always prefixed `evt_`.

#### `proposal_to_entry(proposal, project_root) -> CatalogEntry`

```rust
fn proposal_to_entry(proposal: &ProposedEvent, project_root: &Path) -> CatalogEntry {
    let rel_path = proposal
        .source_path
        .strip_prefix(project_root)
        .unwrap_or(&proposal.source_path)
        .to_string_lossy()
        .to_string();

    let kind_str = match proposal.kind {
        EventKind::PageView   => "pageView",
        EventKind::ApiCall    => "apiCall",
        EventKind::AuthEvent  => "authEvent",
        EventKind::FormSubmit => "formSubmit",
        EventKind::Error      => "error",
    };

    let id = generate_stable_id(&proposal.name, &rel_path, kind_str);

    let provenance = vec![EventProvenance {
        source_path: rel_path,
        line: None,
        adapter: String::new(),
    }];

    let properties = proposal
        .properties
        .iter()
        .map(|h| EventProperty {
            name: h.name.clone(),
            prop_type: h.type_hint.clone(),
            required: false,
            pii: h.pii_hint,
        })
        .collect();

    CatalogEntry {
        id,
        name: proposal.name.clone(),
        description: String::new(),
        status: EventStatus::Proposed,
        confidence: f64::from(proposal.confidence),
        kind: event_kind_to_catalog(proposal.kind),
        provenance,
        properties,
        providers: Vec::new(),
    }
}
```

Private helper. `strip_prefix` gracefully falls back to the full path if the proposal's source is not under `project_root` (e.g., in tests with relative paths).

#### `pub fn from_proposals(proposals: &[ProposedEvent], project_root: &Path) -> Catalog`

```rust
pub fn from_proposals(proposals: &[ProposedEvent], project_root: &Path) -> Catalog {
    let mut events: Vec<CatalogEntry> = proposals
        .iter()
        .map(|p| proposal_to_entry(p, project_root))
        .collect();

    // Stable sort by ID for diff-friendly YAML output.
    events.sort_by(|a, b| a.id.cmp(&b.id));
    // Deduplicate: proposals for the same (name, path, kind) produce the same ID.
    events.dedup_by(|a, b| a.id == b.id);

    Catalog {
        schema_version: CATALOG_SCHEMA_VERSION,
        events,
    }
}
```

**Edge cases:**
- Empty proposals → `Catalog { schema_version: 1, events: [] }`.
- Duplicate proposals (same name+path+kind) → deduplicated to one entry.

#### `pub fn merge_proposals(catalog: &mut Catalog, proposals: &[ProposedEvent], project_root: &Path)`

```rust
pub fn merge_proposals(
    catalog: &mut Catalog,
    proposals: &[ProposedEvent],
    project_root: &Path,
) {
    let existing_ids: std::collections::HashSet<&str> =
        catalog.events.iter().map(|e| e.id.as_str()).collect();

    let new_entries: Vec<CatalogEntry> = proposals
        .iter()
        .map(|p| proposal_to_entry(p, project_root))
        .filter(|e| !existing_ids.contains(e.id.as_str()))
        .collect();

    catalog.events.extend(new_entries);
    catalog.events.sort_by(|a, b| a.id.cmp(&b.id));
}
```

**Key invariant:** Existing entries (including those with `Approved` or `Ignored` status and any human edits to name/description/properties/providers) are NEVER modified. Only genuinely new event IDs are appended. Sorting is re-applied after merge to maintain stable output.

**Edge cases:**
- All proposals already in catalog → no changes.
- Catalog is empty → equivalent to `from_proposals`.
- Proposal whose name was renamed by the reviewer (different name, same original source/kind) → gets a new ID (different name → different hash) → added as a second proposed entry. This is intentional; E4.1 (incremental re-scan) will handle the three-way merge.

#### `pub fn load_catalog(path: &Path) -> Result<Catalog>`

```rust
pub fn load_catalog(path: &Path) -> Result<Catalog> {
    let text = std::fs::read_to_string(path).map_err(Error::Io)?;
    serde_yaml::from_str(&text).map_err(|e| Error::CatalogParse {
        path: path.to_path_buf(),
        message: e.to_string(),
    })
}
```

**Error cases:**
- `Error::Io` if the file does not exist or is not readable.
- `Error::CatalogParse` if the YAML is malformed or the schema does not match `Catalog`.

#### `pub fn save_catalog(catalog: &Catalog, path: &Path) -> Result<()>`

```rust
pub fn save_catalog(catalog: &Catalog, path: &Path) -> Result<()> {
    // Ensure stable sort before writing.
    let mut sorted = catalog.clone();
    sorted.events.sort_by(|a, b| a.id.cmp(&b.id));

    let text = serde_yaml::to_string(&sorted).map_err(|e| Error::CatalogParse {
        path: path.to_path_buf(),
        message: format!("serialize: {e}"),
    })?;

    if let Some(parent) = path.parent()
        && !parent.as_os_str().is_empty()
    {
        std::fs::create_dir_all(parent).map_err(Error::Io)?;
    }
    std::fs::write(path, text).map_err(Error::Io)?;
    Ok(())
}
```

**Edge cases:**
- Parent dir does not exist → `create_dir_all` creates it (same pattern as `Config::save`).
- Catalog is empty → writes a valid YAML with `schemaVersion: 1\nevents: []\n`.
- Sort is applied to the clone before serializing, so the caller's in-memory `catalog` is not mutated.

**Dependencies:** `infergen-types` types (File 3), `crate::Error::CatalogParse` (File 5), `crate::{ProposedEvent, adapter::EventKind}`.
