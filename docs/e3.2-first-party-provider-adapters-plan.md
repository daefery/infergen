## Scope & Design Decision

E3.2 extends the TypeScript **code generator** to emit ready-to-use provider adapter classes. When a user's `infergen.config.*` lists providers, `infergen generate` appends adapter class definitions to the generated SDK file.

**Why codegen, not Rust `ProviderPlugin` impls?**
The generated `.ts` file IS the runtime SDK shipped in the user's project (PRD §6.4). Providers are TypeScript classes the user's app instantiates. Rust-side HTTP clients would be a separate server-side tool (not yet needed; E3.3 handles delivery). Codegen keeps the project dependency-free (no `reqwest`/`ureq`) and fully testable via string assertions.

**Provider transport:** Native `fetch()` calling each service's HTTP ingestion API. Available in all modern browsers and Node.js 18+. No third-party SDKs required — zero additional `npm install`.

**No hardcoded secrets:** Generated classes accept options at construction time. Users pass their API keys: `new PostHogProvider({ apiKey: process.env.POSTHOG_API_KEY })`.

**Determinism:** Provider adapter code is determined solely by which provider names appear in the config — not by the config values. Output is byte-identical for the same provider list.

## Provider HTTP APIs

| Provider | Endpoint | Auth | Key config field |
|---|---|---|---|
| PostHog | `POST {host}/capture/` | body `api_key` | `host` (default `https://us.i.posthog.com`) |
| Segment | `POST {host}/v1/track` | Basic `{writeKey}:` | `host` (default `https://api.segment.io`) |
| Amplitude | `POST https://api2.amplitude.com/2/httpapi` | body `api_key` | — |
| Mixpanel | `POST https://api.mixpanel.com/track` | body `token` | — |
| GA4 | `POST https://www.google-analytics.com/mp/collect?measurement_id=…&api_secret=…` | query params | — |
| RudderStack | `POST {dataPlaneUrl}/v1/track` | Basic `{writeKey}:` | `dataPlaneUrl` required |
| HttpWebhook | `POST {url}` | none (or custom `Authorization` header) | `url` required |

**New private function `write_provider_adapters`:**

```rust
fn write_provider_adapters(out: &mut String, providers: &[ProviderConfig]) {
    out.push('\n');
    out.push_str("// ---------------------------------------------------------------------------\n");
    out.push_str("// Provider Adapters — generated from infergen.config.*\n");
    out.push_str("// ---------------------------------------------------------------------------\n");
    for p in providers {
        match p.name.as_str() {
            "posthog"     => write_posthog_adapter(out),
            "segment"     => write_segment_adapter(out),
            "amplitude"   => write_amplitude_adapter(out),
            "mixpanel"    => write_mixpanel_adapter(out),
            "ga4"         => write_ga4_adapter(out),
            "rudderstack" => write_rudderstack_adapter(out),
            "webhook"     => write_webhook_adapter(out),
            _             => write_unknown_provider_comment(out, &p.name),
        }
    }
}
```

**PostHog:**
```typescript

// ─── PostHog ──────────────────────────────────────────────────────────────
// Docs: https://posthog.com/docs/api/capture

/** Construction options for PostHogProvider. */
export interface PostHogProviderOptions {
  /** PostHog project API key (starts with "phc_" or "ph-"). */
  apiKey: string;
  /** PostHog ingestion host. Defaults to "https://us.i.posthog.com". */
  host?: string;
}

/**
 * PostHog analytics provider adapter.
 * Uses the PostHog Capture HTTP API — no SDK dependency required.
 */
export class PostHogProvider implements Provider {
  readonly id = "posthog";
  constructor(private readonly options: PostHogProviderOptions) {}
  track(eventName: string, properties: Record<string, unknown>): void {
    const host = this.options.host ?? "https://us.i.posthog.com";
    fetch(`${host}/capture/`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        api_key: this.options.apiKey,
        event: eventName,
        properties,
        timestamp: new Date().toISOString(),
      }),
    }).catch(() => {});
  }
}
```

export interface SegmentProviderOptions {

  host?: string;

  track(eventName: string, properties: Record<string, unknown>): void {

      method: "POST",
      headers: {
        "Content-Type": "application/json",
        Authorization: `Basic ${creds}`,
      },
      body: JSON.stringify({
        type: "track",
        anonymousId: this.options.anonymousId ?? "anonymous",
        event: eventName,
        properties,
        timestamp: new Date().toISOString(),
      }),
    }).catch(() => {});
  }
}

/** Construction options for Ga4Provider. */
export interface Ga4ProviderOptions {

  /** Full URL to POST events to. */
  url: string;
  /** Optional additional headers (e.g. Authorization). */

        properties,
        timestamp: new Date().toISOString(),

    }).catch(() => {});

### 2. `crates/infergen-cli/src/commands/generate.rs`

**Purpose:** Load `infergen.config.*` to populate `CodegenConfig.providers`.

**Change:** Load project config from the current directory (or the catalog file's parent) and pass its providers to `CodegenConfig`.

```rust
use infergen_core::{Catalog, CodegenConfig, Config, EventStatus, generate_typescript, load_catalog};

pub fn run(args: GenerateArgs) -> anyhow::Result<()> {
    let catalog = if args.catalog.exists() {
        load_catalog(&args.catalog)
            .with_context(|| format!("reading catalog {}", args.catalog.display()))?
    } else {
        Catalog::default()
    };

    // Load project config to pick up configured providers (best-effort).
    let project_config = args.catalog
        .parent()
        .and_then(|p| p.parent())  // .infergen/ → project root
        .and_then(|root| Config::load_from_dir(root).ok())
        .or_else(|| std::env::current_dir().ok()
            .and_then(|d| Config::load_from_dir(&d).ok()))
        .unwrap_or_default();

    let config = CodegenConfig {
        include_proposed: args.include_proposed,
        providers: project_config.providers,
    };
    // ... rest unchanged
```

**Edge cases:**
- No config file → `Config::default()` → empty `providers` → no adapter section. ✓
- Config file malformed → `ok()` returns `None` → fallback to default. ✓
- `--catalog` is at a non-standard path → falls back to `current_dir()` discovery. ✓

## Open Questions / Risks

- **`btoa()` availability in Node.js < 16:** `btoa` was added to Node.js global in v16. For Node 14 support, use `Buffer.from(...).toString('base64')`. Mitigated: PRD §7 "Portability" implies modern runtimes; Node 18+ is the target. Comment in generated code notes this.
- **`Date.now()` in generated source:** Used inside runtime function bodies (Amplitude `time: Date.now()`). This is a runtime call, not a build-time value — the generated text contains the literal string `Date.now()`, not a timestamp. Determinism is preserved. ✓
- **`full_pipeline_output_contains_no_current_year` test:** Still passes — static generated text doesn't embed `2026-`. The `new Date().toISOString()` string in the generated source does not contain a hardcoded year. ✓
- **`generate_check_up_to_date_exits_zero` test (E2.3):** This test generates then `--check`s. After E3.2, both generate calls in the test produce the same output (no config file in test's tempdir → no providers → same as before). ✓
- **`ProviderConfig` access from `codegen/mod.rs`:** Uses `use crate::config::ProviderConfig;` — intra-crate, no circular dependency. ✓
- **`CodegenConfig` loses `#[derive(Default)]`:** After adding `Vec<ProviderConfig>`, Default still derives cleanly (Vec defaults to empty). ✓
