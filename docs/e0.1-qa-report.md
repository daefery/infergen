## Per-failure detail

### FAIL — Case 21: Naming consistency / git state

**Steps:**
```bash
git status --short
git show HEAD:crates/telemetra-types/Cargo.toml | head -2
head -2 crates/infergen-types/Cargo.toml
```

**Expected:** Committed state and working tree both use consistent naming (either `infergen-*` or `telemetra-*`); working tree is clean.

**Actual:**
```
 M CONTRIBUTING.md
 M Cargo.lock
 M Cargo.toml
 M PRD.md
 M README.md
 M ROADMAP.md
 D crates/telemetra-cli/Cargo.toml
 D crates/telemetra-cli/src/main.rs
 D crates/telemetra-cli/tests/cli.rs
 D crates/telemetra-core/Cargo.toml
 D crates/telemetra-core/src/lib.rs
 D crates/telemetra-types/Cargo.toml
 D crates/telemetra-types/src/lib.rs
 M docs/e0.1-project-scaffold-monorepo-plan.md
 M package.json
 M packages/runtime/package.json
?? .claude/
?? crates/infergen-cli/
?? crates/infergen-core/
?? crates/infergen-types/
```

Committed `HEAD` has `name = "telemetra-types"` in `crates/telemetra-types/Cargo.toml`. Working tree has `name = "infergen-types"` in `crates/infergen-types/Cargo.toml`. The implementation was delivered as `telemetra-*`; the plan specifies `infergen-*`. The user's hooks/tooling subsequently renamed files and package names to `infergen-*` but these changes were not committed.

**Suspected cause:** The implementation in `/dapprove` used `telemetra-*` naming (matching docs at that time), but the plan file and supporting docs (`ROADMAP.md`, `CONTRIBUTING.md`, etc.) were using `infergen-*` naming. Post-commit hooks or the user manually renamed the crate directories and modified package names. The resulting working-tree changes were not staged/committed.

### FAIL — Case 24: release.yml tag trigger pattern

**Steps:**
```bash
grep "tags:" .github/workflows/release.yml
```

**Expected (per plan):** Triggers on tags matching `v*`

**Actual:**
```yaml
on:
  pull_request:
  push:
    tags:
      - '**[0-9]+.[0-9]+.[0-9]+*'
```

**Suspected cause:** `cargo-dist` generates its own tag pattern (`**[0-9]+.[0-9]+.[0-9]+*`) rather than the simpler `v*` specified in the plan. The plan's spec said "Commit the generated file unmodified" — the generated file was committed unmodified, but the generated pattern differs from the plan's stated acceptance criterion. This is a spec gap: the plan described the expected behavior at a high level ("triggers on tags matching `v*`") while the actual cargo-dist output uses a more permissive SemVer pattern. Functionally, both `v0.0.1` and `0.0.1` would trigger.
