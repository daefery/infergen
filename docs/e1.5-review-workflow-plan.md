## PRD Grounding

- **§5 step 4**: "Dev reviews the diff, edits names/properties/providers, marks false positives as `ignored`." — drives mutation functions and CLI commands.
- **§5 step 8**: "Re-run `scan`. New events merge in as proposals; manual edits are preserved." — the diff capability shows what a re-scan added/removed relative to the existing catalog.
- **§6.2**: "status (`proposed` / `approved` / `ignored`)" — `EventStatus` enum already exists in `infergen-types`; E1.5 adds the library functions that mutate status.
- **§6.5**: "Re-scan merges new proposals; preserves manual edits" — `diff_catalogs` makes the delta visible before accepting it.

## Open Questions / Risks

- **Property edit UX**: `upsert_property` and `remove_property` are exposed in the core API but the CLI has no corresponding sub-commands (scope creep risk). CLI only ships the five named sub-commands (list, approve, ignore, rename, describe, diff). Property edits remain YAML-edit-in-editor for now; the library functions exist for future CLI/web use.
- **`diff` without a live scan**: `infergen review diff` requires an explicit `--proposed` path to a second catalog file. The scan command is not wired in CLI until a later epic. This is intentional — diff works today on two YAML files.
- **Name validation**: `rename` validates non-empty (after trim) only. Convention linting is left to `lint_catalog` (E1.3). No redundant validation here.

#### Exported types

```rust
/// A change detected between an existing catalog entry and the proposed re-scan.
#[derive(Debug, Clone, PartialEq)]
pub enum EntryChange {
    NameChanged { from: String, to: String },
    KindChanged { from: CatalogEventKind, to: CatalogEventKind },
    PropertyAdded(EventProperty),
    PropertyRemoved(String),
    PropertyChanged { name: String, from: EventProperty, to: EventProperty },
}

/// An entry present in both catalogs with at least one detected change.
#[derive(Debug, Clone)]
pub struct DiffEntry {
    pub id: String,
    pub existing: CatalogEntry,
    pub proposed: CatalogEntry,
    pub changes: Vec<EntryChange>,
}

/// Structured difference between an existing catalog and a fresh proposed catalog.
///
/// `removed` contains only non-`Ignored` entries so silently-excluded events
/// do not surface as "missing" noise on every re-scan.
#[derive(Debug, Default)]
pub struct CatalogDiff {
    /// Events present in `proposed` but absent from `existing` (new detections).
    pub added: Vec<CatalogEntry>,
    /// Events present in `existing` (status != Ignored) but absent from `proposed`.
    pub removed: Vec<CatalogEntry>,
    /// Events present in both but with at least one changed field.
    pub modified: Vec<DiffEntry>,
    /// Events present in both with no detected change.
    pub unchanged: Vec<CatalogEntry>,
}

impl CatalogDiff {
    /// `true` if added, removed, and modified are all empty.
    pub fn is_clean(&self) -> bool {
        self.added.is_empty() && self.removed.is_empty() && self.modified.is_empty()
    }
}
```

#### Mutation functions

All find the entry by `id` (linear scan — catalogs are small) and return `Error::EventNotFound { id: id.to_owned() }` when not found.

```rust

/// - `EventNotFound` if `id` is not present.
```

#### `diff_catalogs`

```rust
/// Compute the structured difference between `existing` and `proposed`.
///
/// Keyed on stable ID. `proposed` is typically built by `catalog::from_proposals`.
/// `existing` is the current on-disk catalog.
#[must_use]
pub fn diff_catalogs(existing: &Catalog, proposed: &Catalog) -> CatalogDiff
```

Algorithm:
1. Build `existing_map: HashMap<&str, &CatalogEntry>` keyed by ID.
2. Build `proposed_map: HashMap<&str, &CatalogEntry>` keyed by ID.
3. **Added**: entries in `proposed_map` whose ID is absent from `existing_map`.
4. **Removed**: entries in `existing_map` whose ID is absent from `proposed_map` AND `existing.status != EventStatus::Ignored`.
5. **Modified / Unchanged**: entries present in both — call `compute_changes(existing_entry, proposed_entry)`:
   - Compare `name`: if different, push `EntryChange::NameChanged`.
   - Compare `kind`: if different, push `EntryChange::KindChanged`.
   - Build `existing_props: HashMap<&str, &EventProperty>` by name.
   - Build `proposed_props: HashMap<&str, &EventProperty>` by name.
   - For each prop in `proposed_props` not in `existing_props`: `PropertyAdded`.
   - For each prop in `existing_props` not in `proposed_props`: `PropertyRemoved`.
   - For each prop in both: if `prop_type` or `pii` differs: `PropertyChanged`.
   - If `changes` is empty → `unchanged`; else → `modified`.

**Tests (unit)**:
- `diff_empty_catalogs` — both empty → `is_clean()`
- `diff_added_event` — new entry in proposed
- `diff_removed_non_ignored` — existing entry gone, status != Ignored → in removed
- `diff_ignored_not_removed` — existing entry gone but Ignored → not in removed
- `diff_unchanged_entry` — same entry in both → unchanged
- `diff_name_changed` — same ID, different name → modified with NameChanged
- `diff_property_added` — same ID, proposed has extra property → modified with PropertyAdded
- `diff_property_removed` — same ID, existing has extra property not in proposed → modified with PropertyRemoved
- `diff_property_type_changed` — same ID, same prop name, different type → PropertyChanged

**Tests (mutation unit)**:
- `approve_sets_status` — Proposed → Approved
- `approve_idempotent` — Approved → Approved, still Ok
- `ignore_sets_status`
- `approve_unknown_id_errors`

- `rename_empty_name_errors` — `InvalidEventName`

**`Diff`**:
```rust

```

`print_diff` output format:
```
=== Catalog Diff ===
Added (N):
  + <name>  [<kind>]  <source_path>
Removed (N):
  - <name>  [<kind>]  <source_path>
Modified (N):
  ~ <name>  [<kind>]
      name: <from> → <to>
      + property: <name> (<type>)
      - property: <name>
      ~ property: <name>  <from_type> → <to_type>
Unchanged: N events
```
If `diff.is_clean()`, print `"catalog is up to date (no changes detected)"`.

**Private helpers**:
- `load_catalog_or_empty(path: &Path) -> anyhow::Result<Catalog>` — loads if exists, returns empty Catalog if not (for `list` on new projects).
- `parse_status_filter(s: &str) -> anyhow::Result<StatusFilter>` — parses "all", "proposed", "approved", "ignored"; returns error on unknown.
- `status_str(s: EventStatus) -> &'static str` — "proposed", "approved", "ignored".
- `kind_str(k: CatalogEventKind) -> &'static str` — "pageView", "formSubmit", etc.

**Error propagation**: `review::approve(...)` returns `Error::EventNotFound` — propagate via `?`. Clap will print the error message.
