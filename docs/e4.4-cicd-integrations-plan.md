## Scope

Two concrete deliverables:

1. **CLI: `infergen check --json`** — machine-parseable output for CI automation. Adds a `--json` flag to `infergen check` that emits a structured `CheckReport` JSON object to stdout. Exit code unchanged (non-zero when issues found). Enables CI scripts and PR comment bots to parse check results without screen-scraping.

2. **CI/CD recipe templates** — ready-to-paste configuration files for GitHub Actions, GitLab CI, and CircleCI. Each recipe installs infergen, runs `infergen check`, and (for GitHub Actions) posts a formatted PR comment with the check result.

### File 3: `crates/infergen-cli/src/commands/check.rs`

**Purpose:** Implement JSON output mode. When `args.json` is true, serialize a `CheckReport` struct to stdout and skip the human-readable lines. Exit code unchanged regardless of mode.

**New type — `CheckReport`:**
```rust
use serde::Serialize;

/// Structured output of `infergen check --json`.
#[derive(Debug, Serialize)]
pub struct CheckReport {
    pub ok: bool,
    pub issue_count: usize,
    pub new_untracked: Vec<EventRef>,
    pub unreviewed: Vec<EventRef>,
    pub violations: Vec<ViolationRef>,
}

#[derive(Debug, Serialize)]
pub struct EventRef {
    pub id: String,
    pub name: String,
}

#[derive(Debug, Serialize)]
pub struct ViolationRef {
    pub event_name: String,
    pub message: String,
    pub suggestion: Option<String>,
}
```

**Output contract:**
- `ok: true` — catalog is clean, `issue_count: 0`, all arrays empty
- `ok: false` — at least one issue, `issue_count` = sum of all arrays
- Always valid JSON; always exits non-zero when `!ok`

**`run()` modification:**

After computing `new_untracked`, `unreviewed`, `violations`, and `issue_count`:

```rust
if args.json {
    let report = CheckReport {
        ok: issue_count == 0,
        issue_count,
        new_untracked: new_untracked.iter().map(|e| EventRef {
            id: e.id.clone(),
            name: e.name.clone(),
        }).collect(),
        unreviewed: unreviewed.iter().map(|e| EventRef {
            id: e.id.clone(),
            name: e.name.clone(),
        }).collect(),
        violations: violations.iter().map(|v| ViolationRef {
            event_name: v.event_name.clone(),
            message: v.message.clone(),
            suggestion: v.suggestion.clone(),
        }).collect(),
    };
    println!("{}", serde_json::to_string_pretty(&report)?);
    if issue_count > 0 {
        anyhow::bail!("");  // exit non-zero; message suppressed (JSON already has details)
    }
    return Ok(());
}
```

**`anyhow::bail!("")` note:** When `--json` is set and there are issues, we must exit non-zero (required by CI gate). The empty bail message keeps stderr clean — the JSON on stdout is the authoritative output. CI scripts should not depend on stderr content in `--json` mode.

**Imports:** Add `use serde_json;` at top (already `use serde::Serialize` via the struct derive).

**Error handling:** `serde_json::to_string_pretty` returns `Result`; propagated via `?`. Serialization of these simple types cannot fail in practice, but we propagate for correctness.

**Dependencies:** `serde_json` (new to this crate), `serde::Serialize` (via `infergen-core` / workspace `serde` dep, which is already in scope).

Wait — `serde` derive needs to be available in `infergen-cli`. `infergen-core` uses it but `infergen-cli` may not have `serde` itself. Check: `cli.rs` doesn't currently use serde. Solution: add `serde = { workspace = true }` to `infergen-cli/Cargo.toml` alongside `serde_json`.

### File 4: `docs/ci-recipes/github-actions.yml`

**Purpose:** Complete, ready-to-paste GitHub Actions workflow that runs `infergen check` on every push/PR and posts a structured PR comment. This is a recipe file for user projects.

**Full content:**
```yaml
# Infergen Analytics Check — GitHub Actions recipe
#
# Copy this file to .github/workflows/infergen.yml in your project.
#
# Requirements:
#   - infergen binary available (see "Install infergen" step below)
#   - GITHUB_TOKEN secret (automatically provided by GitHub Actions)
#
# The workflow:
#   1. Runs `infergen check` to detect untracked moments, unreviewed events,
#      and naming convention violations.
#   2. On pull requests: posts a formatted PR comment with the check result.

name: Infergen Analytics Check

on:
  push:
    branches: [main]
  pull_request:

concurrency:
  group: infergen-${{ github.ref }}
  cancel-in-progress: true

jobs:
  infergen-check:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write   # required to post PR comments

    steps:
      - uses: actions/checkout@v4

      # ── Install infergen ────────────────────────────────────────────────────
      #
      # Option A (recommended for projects that build from source or during
      # development): install from a release binary via cargo-dist's installer.
      # Replace the URL with your infergen release URL once published.
      #
      # Option B (shown below): install via `cargo install` from crates.io.
      # This is slower but requires no binary hosting.
      #
      - name: Install Rust toolchain
        uses: dtolnay/rust-toolchain@stable

      - name: Cache cargo registry
        uses: Swatinem/rust-cache@v2
        with:
          shared-key: infergen-check

      - name: Install infergen
        # Once infergen is published to crates.io, this is the canonical install.
        # For pre-release: cargo install --git https://github.com/daefery/infergen
        run: cargo install infergen

      # ── Run check ──────────────────────────────────────────────────────────
      - name: Run infergen check
        id: infergen
        run: |
          infergen check --json > infergen-check.json
          echo "exit_code=$?" >> "$GITHUB_OUTPUT"
        continue-on-error: true

      # ── Post PR comment ─────────────────────────────────────────────────────
      - name: Post PR comment
        if: github.event_name == 'pull_request'
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          PR_NUMBER: ${{ github.event.pull_request.number }}
        run: |
          set -euo pipefail
          RESULT=$(cat infergen-check.json)
          OK=$(echo "$RESULT" | jq -r '.ok')
          ISSUE_COUNT=$(echo "$RESULT" | jq -r '.issue_count')

          if [ "$OK" = "true" ]; then
            COMMENT="## Infergen Analytics Check ✅\n\nCatalog is clean — no untracked moments, unreviewed events, or naming violations."
          else
            COMMENT="## Infergen Analytics Check ❌ — $ISSUE_COUNT issue(s) found\n"

            NEW=$(echo "$RESULT" | jq -r '.new_untracked[] | "- \(.name) (`\(.id)`)"')
            UNREVIEWED=$(echo "$RESULT" | jq -r '.unreviewed[] | "- \(.name) (`\(.id)`)"')
            VIOLATIONS=$(echo "$RESULT" | jq -r '.violations[] | "- **\(.event_name)**: \(.message)\(if .suggestion then " → `\(.suggestion)`" else "" end)"')

            if [ -n "$NEW" ]; then
              COMMENT="${COMMENT}\n\n### New untracked moments\nRun \`infergen scan\` then \`infergen review\` to record these:\n${NEW}"
            fi
            if [ -n "$UNREVIEWED" ]; then
              COMMENT="${COMMENT}\n\n### Unreviewed events\nRun \`infergen review\` to approve or ignore:\n${UNREVIEWED}"
            fi
            if [ -n "$VIOLATIONS" ]; then
              COMMENT="${COMMENT}\n\n### Naming convention violations\n${VIOLATIONS}"
            fi
          fi

          printf '%b' "$COMMENT" | gh pr comment "$PR_NUMBER" --body-file -

      # ── Fail the job if check failed ────────────────────────────────────────
      - name: Fail on check issues
        if: steps.infergen.outputs.exit_code != '0'
        run: |
          echo "infergen check failed — see PR comment for details"
          exit 1
```

**Key design decisions:**
- `continue-on-error: true` on the check step so the comment step can still run on failure
- `exit_code` output used to fail the job after the comment is posted
- `jq` is pre-installed on `ubuntu-latest`
- `printf '%b'` interprets `\n` escapes without `echo -e`
- `--body-file -` reads from stdin (safe for multi-line bodies)

**Dependencies:** `jq` (pre-installed on `ubuntu-latest`), `gh` CLI (pre-installed on GitHub Actions runners).

**Tests:** Recipe is a template — no automated tests. Correctness verified by review of GitHub Actions syntax and `jq` expressions. Documented in QA as N/A (template file).

# Merge this snippet into your project's .gitlab-ci.yml.
# Requires: infergen binary on PATH (see "Install infergen" step below).

infergen-check:
  stage: test
  image: rust:1.85
  variables:
    CARGO_HOME: "$CI_PROJECT_DIR/.cargo"
  cache:
    key: "infergen-cargo-$CI_COMMIT_REF_SLUG"
    paths:
      - .cargo/bin/
      - .cargo/registry/
  before_script:
    # Install infergen (cached across runs via .cargo/bin/)
    - |
      if [ ! -f "$CARGO_HOME/bin/infergen" ]; then
        cargo install infergen
      fi
    - export PATH="$CARGO_HOME/bin:$PATH"
  script:
    - infergen check
  # Optional: output JSON for downstream steps
  artifacts:
    when: always
    paths:
      - infergen-check.json
    expire_in: 1 week
  # JSON output step (add to script: block to also produce machine-readable output)
  # - infergen check --json | tee infergen-check.json
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

## Open Questions / Risks

1. **`serde` derive in `infergen-cli`:** The crate doesn't currently use serde directly. Adding `serde = { workspace = true }` to `Cargo.toml` enables `#[derive(Serialize)]` on the local types. Low risk — serde is already compiled transitively.

2. **Recipe files are untested in CI:** The YAML templates can't be automatically executed in the infergen test suite. They are verified by code review and local inspection. QA will document them as N/A with rationale.

3. **`cargo install infergen` not yet possible:** infergen is not published to crates.io. Recipes use `cargo install infergen` with a note to use `--git` or binary download until published. E8.3 resolves this.

4. **PR comment deduplication:** The GitHub recipe posts a new comment on every run. It doesn't update an existing comment. This is acceptable for an initial recipe; a production action might use `gh pr comment --edit-last`. Left as a comment in the recipe.
